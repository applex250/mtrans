# ABSTRACT

本白皮书提出了 OpenFlow：这是一种让研究人员在其日常使用的网络中运行实验协议的方法。OpenFlow 基于具有内部流表的以太网交换机，并提供了一个用于添加和删除流表项的标准化接口。我们的目标是鼓励网络厂商在其交换机产品中添加 OpenFlow，以便在大学校园骨干网和配线间中进行部署。我们认为 OpenFlow 是一个务实的折中方案：一方面，它允许研究人员以统一的方式、在异构交换机上以线速和高端口密度运行实验；另一方面，厂商无需暴露其交换机的内部工作原理。除了允许研究人员在真实流量环境中评估他们的想法外，OpenFlow 还可以作为提议中的大规模试验床（如 GENI）中有用的校园组件。斯坦福大学有两栋大楼将很快使用商用以太网交换机和路由器运行 OpenFlow 网络。我们将努力促进其在其他学校的部署；同时，我们也鼓励您考虑在您所在的大学网络中部署 OpenFlow。

# 1. 可编程网络的需求

网络已成为我们企业、家庭和学校关键基础设施的一部分。对于网络研究人员而言，这种成功既是福音也是诅咒；他们的工作变得更加切题，但产生实际影响的机会却更加渺茫。任何特定的网络创新在现实世界中影响力的减弱，是由于庞大的设备和协议存量，以及不愿尝试在生产流量上进行实验，这为新思想的进入创造了一个极高的门槛。如今，几乎没有实际可行的方法能够在足够逼真的环境（例如，承载真实流量的规模环境）中尝试新的网络协议（例如，新的路由协议或IP的替代方案），从而获得广泛部署所需的信心。结果是，网络研究界提出的大多数新思想都未经尝试和测试；因此人们普遍认为网络基础设施已经“僵化”。

认识到这个问题后，网络界正致力于开发可编程网络，例如GENI [1]，这是一个提议中的全国性研究设施，用于实验新的网络架构和分布式系统。这些可编程网络需要可编程的交换机和路由器（使用虚拟化）能够同时处理多个隔离实验网络的数据包。例如，在GENI中，预计研究人员将被分配整个网络中的一片资源，由部分网络链路、数据包处理单元（例如路由器）和端主机组成；研究人员可以按照自己的意愿对切片进行编程。切片可以跨越骨干网，延伸到接入网，进入大学校园、工业研究实验室，并包括配线间、无线网络和传感器网络。

虚拟化可编程网络可以降低新思想的准入门槛，提高网络基础设施的创新速度。但是，建设全国性设施的计划非常宏大（且成本高昂），并且需要数年时间才能部署。

本白皮书聚焦于一个更贴近现实且更短期的问题：作为研究人员，我们要如何在我们的校园网络中运行实验？如果我们能弄清楚方法，我们可以很快开始，并将该技术扩展到其他校园，以造福整个社区。

为了应对这一挑战，需要回答几个问题，包括：在早期阶段，大学网络管理员如何才能放心地将实验设备（交换机、路由器、接入点等）放入他们的网络中？研究人员如何才能控制其本地网络的一部分，而不干扰依赖该网络的其他人？并且，网络交换机究竟需要什么功能才能支持实验？我们的目标是提出一种新的交换机功能，它可以帮助将可编程性扩展到大学校园的配线间。

我们不采取的一种方法是说服商业“名牌”设备供应商在其交换机和路由器上提供一个开放、可编程、虚拟化的平台，以便研究人员可以部署新协议，同时网络管理员也能放心地认为设备具有良好的支持。这种结果在短期内极不可能发生。商业交换机和路由器通常不提供开放的软件平台，更不用说提供虚拟化其硬件或软件的手段了。商业网络的做法是，标准化的外部接口非常狭窄（即仅限于数据包转发），并且交换机的所有内部灵活性都被隐藏了。内部结构因供应商而异，没有供研究人员尝试新思想的标准平台。此外，网络设备供应商对开放其设备内部的接口 understandably nervous（可以理解地感到紧张）：他们花费了数年时间部署和调整脆弱的分布式协议和算法，他们担心新的实验会导致网络崩溃。当然，开放平台也为新的竞争对手降低了准入门槛。

一些开放的软件平台已经存在，但并不具备我们所需的性能或端口密度。最简单的例子是带有多个网络接口和操作系统的PC。所有众所周知的操作系统都支持接口间的数据包路由，并且存在路由协议的开源实现（例如，作为Linux发行版的一部分，或来自XORP [2]）；在大多数情况下，可以修改操作系统以几乎任何方式处理数据包（例如，使用Click [3]）。当然，问题是性能：PC既无法支持大学配线间所需的端口数量（每台设备需要 $1 0 0 +$ 个端口的扇出能力），也无法支持数据包处理性能（配线间交换机处理超过100Gbits/s的数据，而典型的PC很难超过1Gbit/s；而且两者之间的差距正在扩大）。

现有的具有线速处理专用硬件的平台也不太适合大学配线间。例如，华盛顿大学正在开发一种名为Supercharged PlanetLab Platform [4] 的基于ATCA的虚拟化可编程路由器，它可以使用网络处理器在线速下同时处理来自多个接口的数据包。这种方法从长远来看是有前景的，但目前的目标是大型交换中心，对于在大学配线间广泛部署来说太昂贵了。另一个极端是NetFPGA [5]，旨在用于教学和研究实验室。NetFPGA是一种低成本PCI卡，带有用于处理数据包的用户可编程FPGA，以及4个千兆以太网端口。NetFPGA仅限于四个网络接口——对于配线间使用来说是不够的。

因此，商业解决方案过于封闭和僵化，而研究解决方案要么性能不足或扇出能力不够，要么过于昂贵。具有完全通用性的研究解决方案似乎不太可能克服其性能或成本限制。一种更有前景的方法是在通用性上做出妥协，并寻求一定程度的交换机灵活性，这种灵活性需要：

• 适合高性能和低成本的实现。

• 能够支持广泛的研究。

• 确保将实验流量与生产流量隔离。

# 2. OpenFlow 交换机

基本思想很简单：我们利用这样一个事实，即大多数现代以太网交换机和路由器包含以线速运行的流表（通常由 TCAM 构建），用于实现防火墙、NAT、QoS 以及收集统计数据。虽然每个供应商的流表不同，但我们已经确定了一个有趣的通用功能集，这些功能在许多交换机和路由器中运行。OpenFlow 利用了这个通用功能集。

OpenFlow 提供了一种开放协议，用于对不同的交换机和路由器中的流表进行编程。网络管理员可以将流量划分为生产流量和研究流量。研究人员可以控制自己的流——通过选择其数据包遵循的路由以及它们接收的处理方式。通过这种方式，研究人员可以尝试新的路由协议、安全模型、寻址方案，甚至是 IP 的替代方案。在同一个网络上，生产流量被隔离，并以与今天相同的方式进行处理。

OpenFlow 交换机的数据通路由一个流表和与每个流条目关联的动作组成。OpenFlow 交换机支持的动作集是可扩展的，但在下文中我们描述了所有交换机的最低要求。为了高性能和低成本，数据通路必须具有精心规定的灵活性程度。这意味着放弃指定对每个数据包进行任意处理的能力，并寻求一个更有限但仍然有用的动作范围。因此，在本文后面，我们定义了所有 OpenFlow 交换机的基本必需动作集。

OpenFlow 交换机至少由三部分组成：(1) 流表，每个流条目都有一个关联的动作，告诉交换机如何处理该流；(2) 安全通道，将交换机连接到远程控制进程（称为控制器），允许在控制器和交换机之间发送命令和数据包，使用 (3) OpenFlow 协议，它提供了一种开放且标准的方式让控制器与交换机进行通信。通过指定一个标准接口，即可以通过外部定义流表条目的 OpenFlow 协议，OpenFlow 交换机避免了研究人员对交换机进行编程的需要。

将交换机分类为不支持正常二层和三层处理的专用 OpenFlow 交换机，以及增加了 OpenFlow 协议和接口作为新特性的支持 OpenFlow 的通用商用以太网交换机和路由器，是有用的。

专用 OpenFlow 交换机。专用 OpenFlow 交换机是一个哑数据通路元素，根据远程控制进程的定义在端口之间转发数据包。图 1 显示了 OpenFlow 交换机的一个示例。

在此背景下，流的定义很广泛，仅受流表特定实现能力的限制。例如，流可以是一个 TCP 连接，或者来自特定 MAC 地址或 IP 地址的所有数据包，或者具有相同 VLAN 标签的所有数据包，或者来自同一交换机端口的所有数据包。对于涉及非 IPv4 数据包的实验，流可以定义为匹配特定（但非标准）报头的所有数据包。

每个流条目都有一个与之关联的简单动作；三个基本动作（所有专用 OpenFlow 交换机必须支持）是：

1. 将此流的数据包转发到给定端口（或多个端口）。这允许数据包通过网络进行路由。在大多数交换机中，这预期以线速进行。

2. 封装并将此流的数据包转发给控制器。数据包被传送到安全通道，在那里它被封装并发送给控制器。通常用于新流中的第一个数据包，以便控制器可以决定是否应将该流添加到流表中。或者在某些实验中，它可用于将所有数据包转发给控制器进行处理。

3. 丢弃此流的数据包。可用于安全性，以遏制拒绝服务攻击，或减少来自终端主机的虚假广播发现流量。

流表中的条目有三个字段：(1) 定义流的包头部，(2) 定义应如何处理数据包的动作，以及 (3) 统计数据，它跟踪每个流的数据包数和字节数，以及自上次数据包匹配该流以来的时间（以帮助移除不活动的流）。

在第一代 “Type 0” 交换机中，流头部是表 1 中所示的 10 元组。

支持 OpenFlow 的交换机。一些商用交换机、路由器和接入点将通过添加流表、安全通道和 OpenFlow 协议来增强 OpenFlow 功能（我们在第 5 节中列出了一些示例）。通常，流表将重用现有硬件，如 TCAM；安全通道和协议将被移植到交换机的操作系统上运行。图 2 显示了一个由支持 OpenFlow 的商用交换机和接入点组成的网络。在此示例中，所有流表均由同一控制器管理；OpenFlow 协议允许交换机由两个或更多控制器控制，以提高性能或鲁棒性。

我们的目标是使实验能够在现有的生产网络中与常规流量和应用程序一起进行。因此，为了赢得网络管理员的信任，支持 OpenFlow 的交换机必须将实验流量（由流表处理）与将由交换机的正常二层和三层流水线处理的生产流量隔离开来。有两种方法可以实现这种分离。一种是添加第四个动作：

4. 通过交换机的正常处理流水线转发此流的数据包。

另一种是为实验流量和生产流量定义单独的 VLAN 集。这两种方法都允许不属于实验一部分的正常生产流量以通常的方式由交换机处理。所有支持 OpenFlow 的交换机都需要支持其中一种方法；有些将支持这两种方法。

附加特性。如果交换机支持上述头部格式和四个基本动作（并在 OpenFlow 交换机规范中详述），那么我们称之为 “Type 0” 交换机。我们预计许多交换机将支持附加动作，例如重写数据包头部的一部分（例如，用于 NAT，或在中间链路上混淆地址），以及将数据包映射到优先级类别。同样，一些流表将能够匹配数据包头部中的任意字段，从而能够使用新的非 IP 协议进行实验。随着特定特性集的出现，我们将定义 “Type 1” 交换机。

控制器。控制器代表实验从流表中添加和删除流条目。例如，静态控制器可能是在 PC 上运行的简单应用程序，用于在实验期间静态建立流以互连一组测试计算机。在这种情况下，流类似于当前网络中的 VLAN —— 提供一种简单的机制来将实验流量与生产网络隔离开来。从这个角度看，OpenFlow 是 VLAN 的泛化。

人们也可以想象更复杂的控制器，随着实验的进行动态添加/删除流。在一种使用模型中，研究人员可能控制完整的 OpenFlow 交换机网络，并自由决定所有流如何被处理。更复杂的控制器可能支持多个研究人员，每个研究人员拥有不同的帐户和权限，使他们能够在不同的流集上运行多个独立的实验。被识别为受特定研究人员控制的流（例如，通过在控制器中运行的策略表）可以被传递到研究人员的用户级控制程序，该程序然后决定是否应向交换机网络添加新的流条目。

# 3. 使用 OpenFlow

作为一个关于如何使用 OpenFlow 交换机的简单示例，设想 Amy（一位研究人员）发明了 Amy-OSPF，作为一种用于替代 OSPF 的新路由协议。她希望在不更改任何终端主机软件的情况下，在 OpenFlow 交换机网络中尝试她的协议。Amy-OSPF 将在控制器上运行；每当一个新的应用流启动时，Amy-OSPF 会选择一条穿过一系列 OpenFlow 交换机的路径，并沿该路径在每个交换机中添加一个流表项。在她的实验中，Amy 决定对从她自己的桌面 PC 进入 OpenFlow 网络的流量使用 Amy-OSPF——这样她就不会干扰其他人的网络。为此，她将一个流定义为所有通过连接其 PC 的交换机端口进入 OpenFlow 交换机的流量，并添加一个流表项，动作为“封装所有数据包并转发到控制器”。当她的数据包到达控制器时，她的新协议会选择一条路由，并沿选定路径向每个交换机添加一个新的流表项（针对该应用流）。当后续数据包到达交换机时，它们将由流表快速（且以线速）进行处理。

关于随着实验进展动态添加和删除流的控制器的性能、可靠性和可扩展性，存在一些值得探讨的问题：这样的集中式控制器能否足够快地处理新流并对流交换机进行编程？当控制器发生故障时会发生什么？在某种程度上，这些问题在使用简单流交换机和中央控制器的 Ethane 原型背景下得到了解决 [7]。初步结果表明，基于低成本桌面 PC 的 Ethane 控制器每秒可处理超过 10,000 个新流——足以应对大型大学校园的需求。当然，新流的处理速率将取决于研究人员实验所需的处理复杂度。但这使我们有信心运行有意义的实验。通过使控制器（及实验）无状态，可以实现可扩展性和冗余，从而允许在多个独立设备之间进行简单的负载均衡。

# 3.1 生产网络中的实验

很可能，Amy 正在一个被许多其他人使用的网络中测试她的新协议。因此，我们希望该网络具有两个额外的属性：

1. 属于 Amy 以外用户的分组，应使用来自“知名品牌”供应商的交换机或路由器中运行的标准且经过测试的路由协议进行路由。

2. Amy 应仅能够为她自己的流量，或者为她的网络管理员允许她控制的任何流量添加流表项。

属性 1 由支持 OpenFlow 的交换机实现。在 Amy 的实验中，对于所有并非来自 Amy 的 PC 的分组，其默认操作可能是通过正常处理流水线将其转发。Amy 自己的分组将被直接转发到出端口，而无需经过正常流水线的处理。

属性 2 取决于控制器。控制器应被视为一个使研究人员能够实现各种实验的平台，属性 2 的限制可以通过适当使用权限或其他方式来限制个别研究人员控制流表项的权限来实现。这些类权限机制的具体性质将取决于控制器的实现方式。我们预计将出现各种各样的控制器。作为控制器具体实现的一个示例，部分作者正在开发一个名为 NOX 的控制器，作为 Ethane 工作 [8] 的延续。另一种截然不同的控制器可能会通过将 GENI 管理软件扩展至 OpenFlow 网络而出现。

# 3.2 更多示例

像任何实验平台一样，实验的数量将超过我们预先能想到的——OpenFlow 网络中的大多数实验尚未被构想出来。在此，为了说明，我们提供一些示例，说明如何使用启用 OpenFlow 的网络来试验新的网络应用和架构。

示例 1：网络管理与访问控制。我们将以 Ethane 作为第一个示例 [7]，因为它是启发 OpenFlow 的研究。事实上，OpenFlow 交换机可以被视为 Ethane 数据路径交换机的泛化。Ethane 使用了一个控制器的特定实现，适用于网络管理和控制，管理流的准入和路由。Ethane 的基本思想是允许网络管理者定义一个

在中央控制器中定义的全网策略，该策略通过对每个新流做出准入控制决策来直接执行。控制器根据一组规则检查新流，例如“访客可以使用 HTTP 进行通信，但只能通过 Web 代理”或“VoIP 电话不允许与笔记本电脑通信”。控制器通过管理名称和地址之间的所有绑定，将数据包与其发送者关联起来——它本质上接管了 DNS、DHCP 并在用户加入时对其进行身份验证，跟踪他们连接到哪个交换机端口（或接入点）。可以设想对 Ethane 进行扩展，其中策略规定将特定的流发送到控制器中的用户进程，从而允许在网络中执行特定于研究人员的处理。

示例 2：VLAN。OpenFlow 可以像 VLAN 一样轻松地为用户提供他们自己的隔离网络。最简单的方法是静态声明一组流，指定给定 VLAN ID 上的流量可以访问的端口。被识别为来自单个用户（例如，源自特定的交换机端口或 MAC 地址）的流量由交换机（通过操作）标记为适当的 VLAN ID。

一种更动态的方法可能会使用控制器来管理用户的身份验证，并利用用户位置的知识在运行时对流量进行标记。

示例 3：移动无线 VoIP 客户端。对于此示例，考虑针对支持 WiFi 的手机的新型呼叫切换机制实验。在实验中，VoIP 客户端通过启用 OpenFlow 的网络建立新连接。实现了一个控制器来跟踪客户端的位置，当用户在网络中移动时，通过重新编程流表来重新路由连接，从而允许从一个接入点到另一个接入点的无缝切换。

示例 4：非 IP 网络。到目前为止，我们的示例都假设了一个 IP 网络，但 OpenFlow 不求数据包必须采用任何特定格式——只要流表能够匹配数据包头即可。这将允许使用新的命名、寻址和路由方案进行实验。启用 OpenFlow 的交换机有几种方式可以支持非 IP 流量。例如，可以使用以太网头（MAC 源地址和目的地址）、新的 EtherType 值，或者在 IP 层面使用新的 IP 版本号来识别流。更一般地说，我们希望未来的交换机允许控制器创建通用掩码（offset $^ +$ value $^ +$ mask），从而允许按照研究人员指定的方式处理数据包。

示例 5：处理数据包而非流。上面的示例是涉及流的实验——即控制器在流开始时做出决策的实验。当然，也有一些有趣的实验需要处理每个数据包。例如，检查每个数据包的入侵检测系统、显式拥塞控制机制，或在修改数据包内容时，例如将数据包从一种协议格式转换为另一种协议格式时。

在启用 OpenFlow 的网络中，处理数据包有两种基本方法。第一种也是最简单的方法，是强制一个流的所有数据包都通过控制器。为此，控制器不会向流交换机添加新的流条目——它只是允许交换机默认转发

第二种处理数据包的方法是将它们路由到执行数据包处理的可编程交换机——例如，基于 NetFPGA 的可编程路由器。优点是数据包可以以用户可定义的方式在线速进行处理；图 3 展示了如何做到这一点的示例，其中启用 OpenFlow 的交换机本质上充当配线架，允许数据包到达 NetFPGA。在某些情况下，NetFPGA 板（一种插入 Linux PC 的 PCI 板）可能与启用 OpenFlow 的交换机一起放置在配线间中，或者（更可能）放置在实验室中。

# 5. 部署 OpenFlow 交换机

我们相信，对于网络设备厂商而言，向研究社区销售支持 OpenFlow 的交换机蕴含着良好的市场机遇。成千上万所高校的每栋建筑都包含配备以太网交换机和路由器的配线间，以及遍布校园的无线接入点。

我们正在积极与多家交换机和路由器制造商开展合作，这些制造商通过在现有硬件中实现流表，为其产品添加 OpenFlow 功能；即无需更改硬件。交换机利用其现有处理器运行安全通道软件。

我们发现，网络设备厂商非常乐于接受添加 OpenFlow 功能这一构想。大多数厂商希望在支持研究社区的同时，无需暴露其产品的内部运作机制。

我们正在斯坦福大学计算机科学系和电气工程系部署大规模 OpenFlow 网络。两栋楼内的网络将由运行 OpenFlow 的交换机取代。最终，所有流量都将在 OpenFlow 网络上传输，在生产流量和实验流量于不同 VLAN 上隔离的前提下，由网络管理员进行控制。研究人员将能够控制各自的流量，并具备添加/删除流表项的能力。

我们同样预期研究社区将开发出多种不同的 OpenFlow 交换机。OpenFlow 网站提供了面向多种平台的“Type $0 ^ { \dag }$”参考设计：Linux（软件）、OpenWRT（软件，适用于接入点）以及 NetFPGA（硬件，4个 1GE 端口）。随着社区创建出更多的参考设计，我们将予以发布。我们鼓励开发者参照参考设计来测试其交换机。

网站上发布的所有 OpenFlow 交换机参考实现均为开源，可供商业和非商业用途免费使用。2

# 6. 结论

我们相信，OpenFlow 是一种务实的折中方案，它允许研究人员以统一的方式在异构交换机和路由器上运行实验，而无需供应商公开其产品的内部机制，也无需研究人员编写供应商特定的控制软件。

如果我们在校园内部署 OpenFlow 网络取得成功，我们希望 OpenFlow 将逐渐在其他大学普及，从而增加支持实验的网络数量。我们希望新一代控制软件能够涌现，允许研究人员复用控制器和实验，并在此基础上基于他人的工作继续构建。随着时间的推移，我们希望不同大学的 OpenFlow 网络孤岛将通过隧道和覆盖网络相互连接，或许还可以通过运行在连接各大学的骨干网中的新型 OpenFlow 网络来实现互连。
